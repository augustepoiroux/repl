# AGENTS.md

This repository is a multi-version fork of `leanprover-community/repl`.
Agents working here must treat version history as part of the public API.

## Hard Rules

- Never modify a previously existing `v1...` REPL release branch.
- Never retag, move, delete, or overwrite an existing release tag.
- Even a compatibility-only update for new Lean versions must use a new bumped
  `v1...` branch and new tags.
- The reason is reproducibility: backward patch application can accidentally
  change behavior, even when no new REPL feature was intended.
- Do not run destructive Git commands such as `git reset --hard`, forced branch
  updates, forced pushes, tag deletion, or checkout/reset of user changes unless
  the user explicitly confirms the exact operation.
- If a step is ambiguous, risky, or would rewrite shared history, stop and ask.
- Keep two log files for every maintenance/update session:
  - a command log containing every command and its stdout/stderr;
  - a manually edited notes log where the agent writes what it did, why it did
    it, which files it changed, any bug fixes it had to make, and any manual
    decisions or confirmations.

## Repo Shape

- `lakefile.lean`, `lean-toolchain`: root REPL package and Lean toolchain.
- `REPL/Main.lean`: JSON loop and command/tactic dispatch.
- `REPL/JSON.lean`: input and output JSON protocol.
- `REPL/Frontend.lean`: Lean frontend wrapper, incremental command snapshots,
  and header caching.
- `REPL/Snapshots.lean`: command/proof snapshots, proof-state replay, and
  pickling.
- `REPL/Extract/Declaration.lean`: declaration extraction.
- `test.sh`, `Test.lean`: root regression tests.
- `test/Mathlib`: optional Mathlib integration tests.
- `versioning.md`: StGit workflow for creating the Lean-version patch stack.
- `update_lean_version.sh`: helper to update one Lean-version patch.
- `list_missing_versions.sh`: helper to compare Lean releases with local REPL
  compatibility tags.

## Versioning Model

Each REPL release branch is named like `v1.3.15`. Each supported Lean version is
represented by one commit in that branch's StGit stack, and the commit subject
must match the Lean version, for example `v4.29.1`.

Each commit gets a tag:

```text
v<repl-version>_lean-toolchain-v<lean-version>
```

Examples:

```text
v1.3.15_lean-toolchain-v4.29.0
v1.3.16_lean-toolchain-v4.30.0-rc2
```

When adding new Lean versions, always create a new REPL version branch, even if
there are no new REPL features. A single new REPL branch may add several new Lean
versions.

Some Lean versions do not have a corresponding Mathlib tag. For those Lean
versions, keep root REPL tests enabled and disable Mathlib tests. Do not invent a
Mathlib revision. The update helper detects this and runs tests with
`RUN_MATHLIB=0`. CI performs the same check from `lean-toolchain` before running
`lake exe test` and exports `RUN_MATHLIB=1` only when the matching Mathlib tag
exists.

## Full Update Workflow

This workflow is intentionally conservative. Keep two logs:

- `CMD_LOG`: exact commands and their output;
- `NOTES_LOG`: manually edited notes written by the agent about intent,
  decisions, file changes, bug fixes, and anything out of the ordinary.

For shell-driven maintenance, use names like:

```bash
mkdir -p logs
STAMP=$(date +%Y%m%d_%H%M%S)
CMD_LOG=logs/update-v1.3.16-${STAMP}.commands.log
NOTES_LOG=logs/update-v1.3.16-${STAMP}.notes.md
```

Use `tee -a "$CMD_LOG"` or explicit command logging for commands and outputs.
The notes log is not generated by scripts. The agent must edit it directly while
working. It should start with the usual update recipe being followed, then record
each meaningful action, every file edit, any bug fix made during the update, and
any unexpected situation or confirmation request.

Before doing anything, record this information in the command log and summarize
the starting state manually in the notes log:

```bash
git status --short --branch
git remote -v
git branch --show-current
git rev-parse HEAD
```

If there are tracked changes you do not understand, stop and ask.

### 1. Synchronize Remotes

Fetch all remotes and tags:

```bash
git fetch --all --tags --prune
```

Do not force-update local release branches or tags.

### 2. Rebase `master` and `origin/master` on `upstream/master`

The fork's `master`/`origin/master` must be updated by rebasing the fork-only
commits on top of `upstream/master`. Do not merge upstream into `master`; keep
the fork commits as commits on top of upstream.

Recommended non-destructive pattern:

```bash
git checkout master   # local branch tracking origin/master
git status --short --branch
git rebase upstream/master
```

Resolve conflicts carefully. If conflicts are surprising, stop and ask. After
the rebase, run the relevant root tests.

Updating `origin/master` after a rebase may require moving the remote branch.
Do not do that silently. Ask for explicit confirmation, record the old and new
SHAs in the command log and explain the decision in the notes log, and prefer:

```bash
git push --force-with-lease origin master
```

Only use that command after confirmation.

### 3. Update the README Changelog

Before preparing feature branches and the new release branch, update
`README.md` with a new `v1...` changelog entry.

The entry must mention:

- the new REPL version;
- new Lean versions added;
- REPL features and bug fixes included;
- whether some Lean versions skip Mathlib tests due to missing Mathlib tags.

This README change belongs to the updated fork `master` lineage. Feature
branches below are still rebased on `upstream/master`, not on `master`.

### 4. Rebase Feature Branches in Order

The current feature stack order is:

```text
upstream/master
fix-sorry-ctx
incremental_processing
auto-prefix-optimization
declaration-extraction
add-set-option
```

The base of this whole feature stack is `upstream/master`, not `master` and not
`origin/master`. Rebase each branch on the previous branch in the stack,
preserving the order. Use normal `git rebase`; do not reset or force-update
without confirmation.

Sketch:

```bash
git checkout fix-sorry-ctx
git rebase upstream/master

git checkout incremental_processing
git rebase fix-sorry-ctx

git checkout auto-prefix-optimization
git rebase incremental_processing

git checkout declaration-extraction
git rebase auto-prefix-optimization

git checkout add-set-option
git rebase declaration-extraction
```

After each branch rebase, build or test enough to catch obvious breakage. Record
the old and new branch tips in the command log and summarize the result in the
notes log. If conflict resolution requires dropping or changing behavior, ask
before continuing.

### 5. Refresh `nightly`

After `add-set-option` is correct, refresh `nightly` from it and merge the new
fork master into `nightly`.

The intended maintenance operation is to reset `nightly` to the new
`add-set-option`, then merge the updated fork master (`origin/master`, or local
`master` after it has been confirmed to match `origin/master`) into it. Because
this moves a long-lived branch, do not do it silently. Ask for confirmation,
record the old SHA, and create a backup branch first:

```bash
OLD_NIGHTLY=$(git rev-parse nightly)
git branch backup/nightly-before-<date> "$OLD_NIGHTLY"
git checkout nightly
git reset --hard add-set-option
git merge origin/master
```

Only run that reset after explicit confirmation. Pushing the refreshed `nightly`
may require moving the remote branch; ask before any force push and use
`--force-with-lease` if confirmed.

### 6. Create the New REPL Version Branch

Do not edit an old `v1...` branch. Create a new branch with the bumped REPL
version, following `versioning.md`.

Typical StGit flow:

```bash
git checkout <previous-v1-branch>
stg branch -C <new-v1-branch>
stg pop -a
stg new <new-lean-version> -m <new-lean-version>
stg rebase master
```

Use the integration base expected by `versioning.md` and the current release
process. If using `master` here would omit the refreshed feature stack from
`nightly`, stop and ask before rebasing the version branch.

For each missing Lean version, add one StGit patch whose subject is exactly the
Lean version. Use `./list_missing_versions.sh` to identify candidates and
Mathlib availability.

### 7. Update Each Lean-Version Patch

For each patch:

```bash
./update_lean_version.sh <lean-version>
```

If the Lean version has no Mathlib tag, the helper will disable Mathlib tests
for that patch. You can also force that mode with:

```bash
./update_lean_version.sh <lean-version> --skip-mathlib
```

Review the diff after every patch:

```bash
git diff
lake exe test
stg refresh
```

Only keep changes that are necessary for that Lean version and for the feature
backports that are part of the new REPL release.

### 8. Test the Stack

Use the StGit tester:

```bash
./find_first_failing_commit_parallel.sh
```

Run a smaller `--fast` pass first if useful, but do not rely only on `--fast`
for release validation.

When failures occur, go to the failing patch, fix it, refresh the patch, and
restart testing. Record failing patch names and log paths.

### 9. Tag and Push

Only after the full stack passes, tag and push. The tag script validates that
each commit subject matches the root `lean-toolchain`.

```bash
./tag_and_push_all.sh origin <new-v1-branch> master
```

Check the proposed tags before confirming. Never reuse an existing tag name for
a new commit.

## Maintenance Guidance

- Prefer small, auditable patches.
- Keep version-specific compatibility fixes in the Lean-version patch where they
  are required.
- Keep feature changes on the feature branches and let the release stack apply
  them backward.
- Do not hide failing Mathlib tests by editing expected outputs blindly. First
  determine whether the output change is due to Lean behavior, Mathlib behavior,
  or an unintended REPL behavior change.
- If a Lean version lacks a Mathlib tag, document that in the release notes and
  log, and run root REPL tests only for that version.
- When in doubt, ask for confirmation before changing history, changing the
  supported-version policy, or skipping a test class.
